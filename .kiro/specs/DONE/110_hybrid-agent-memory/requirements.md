# Requirements Document

## Introduction

This feature is Phase 2 of the Agent Memory Architecture (Option 4: Hybrid from the design RFC). It depends on spec 100 (Structured Agent Memory) being implemented first and extends it with three capabilities: milestone summaries, a configurable sliding window, and full history elimination from LLM prompts.

Currently, spec 100 provides structured memory (typed per-agent recall of offers, concessions, open items) plus a hardcoded sliding window of the last 3 raw messages. This works well for short negotiations (8–12 turns), but for longer runs (16+ turns), the sliding window alone cannot capture strategic context from early turns, and the fixed window size may not suit all scenario types.

This spec adds:
1. **Milestone summaries** — at configurable turn intervals, a lightweight LLM call generates a compressed strategic summary of the negotiation from each agent's perspective. These summaries capture key positions, major concessions, unresolved disputes, and overall trajectory.
2. **Configurable sliding window** — the window size becomes a scenario-level parameter (`sliding_window_size`, default 3) so scenario authors can tune how much raw conversational context agents see.
3. **Full history elimination** — after the first milestone summary is generated, the prompt builder stops including any raw history beyond the sliding window. Token cost per turn becomes bounded and predictable regardless of negotiation length.

The feature is toggled independently from structured memory via a "Milestone Summaries" toggle in the Advanced Options UI. Enabling milestone summaries automatically enables structured memory (it is a prerequisite). When disabled, behavior is identical to spec 100.

## Glossary

- **Agent_Memory**: The Pydantic V2 model from spec 100 representing per-agent structured memory (offers, concessions, open items, tactics, red lines, compliance status, turn count).
- **Milestone_Summary**: A compressed strategic summary of the negotiation up to a given turn, generated by a lightweight LLM call. Each agent gets a separate summary from their own perspective, including private reasoning context.
- **Milestone_Interval**: A scenario-level integer parameter controlling how many full turn cycles elapse between milestone summary generations (default 4).
- **Sliding_Window**: The last N raw history entries retained in the prompt for conversational context (tone, phrasing, emotional cues). N is configurable per scenario.
- **Sliding_Window_Size**: A scenario-level integer parameter controlling how many recent raw messages are included in the sliding window (default 3).
- **Milestone_Generator**: The module responsible for making the lightweight LLM call to produce a Milestone_Summary from the current negotiation state and history.
- **Agent_Node**: The backend orchestrator function (`create_agent_node` in `agent_node.py`) that builds prompts and invokes the LLM for each agent turn.
- **NegotiationState**: The LangGraph TypedDict (`state.py`) representing the full runtime state of a negotiation session.
- **NegotiationParams**: The Pydantic model in `scenarios/models.py` defining turn limits, agreement threshold, and execution order for a scenario.
- **NegotiationStateModel**: The Pydantic model (`models/negotiation.py`) used for Firestore persistence and API serialization of session state.
- **Arena_Selector**: Screen 2 of the core user flow where users pick a scenario, view agent cards, toggle hidden variables, and initialize a negotiation.
- **Advanced_Options_Section**: A collapsible UI section on the Arena_Selector page that exposes optional feature toggles such as structured agent memory and milestone summaries.
- **StartNegotiationRequest**: The Pydantic model for the POST `/api/v1/negotiation/start` request body.

## Requirements

### Requirement 1: Scenario-Level Sliding Window and Milestone Configuration

**User Story:** As a scenario author, I want to configure the sliding window size and milestone summary interval per scenario, so that I can tune memory behavior for different negotiation types without code changes.

#### Acceptance Criteria

1. THE NegotiationParams model SHALL include a `sliding_window_size` field of type `int` with a default value of 3 and a minimum value of 1
2. THE NegotiationParams model SHALL include a `milestone_interval` field of type `int` with a default value of 4 and a minimum value of 2
3. WHEN a scenario JSON file omits `sliding_window_size` or `milestone_interval`, THE NegotiationParams model SHALL use the default values without validation errors
4. FOR ALL existing scenario JSON files that do not contain `sliding_window_size` or `milestone_interval` fields, THE scenario loader SHALL parse them without errors (backward compatible)

### Requirement 2: NegotiationState Milestone Fields

**User Story:** As a developer, I want the LangGraph runtime state and Firestore persistence model to carry milestone summary data and the enablement flag, so that milestone summaries are available throughout the orchestration pipeline and persisted across requests.

#### Acceptance Criteria

1. THE NegotiationState TypedDict SHALL include a `milestone_summaries_enabled` field of type `bool`
2. THE NegotiationState TypedDict SHALL include a `milestone_summaries` field of type `dict[str, list[dict[str, Any]]]` to store milestone summaries keyed by agent role, where each entry is a list of summary dicts containing at minimum `turn_number` (int) and `summary` (str)
3. THE `create_initial_state` factory function SHALL initialize `milestone_summaries_enabled` from the value passed by the caller
4. WHEN `milestone_summaries_enabled` is True, THE `create_initial_state` function SHALL also set `structured_memory_enabled` to True (milestone summaries require structured memory)
5. WHEN `milestone_summaries_enabled` is True, THE `create_initial_state` function SHALL initialize `milestone_summaries` with an empty list for each agent role
6. WHEN `milestone_summaries_enabled` is False, THE `create_initial_state` function SHALL initialize `milestone_summaries` as an empty dict
7. THE NegotiationState TypedDict SHALL include `sliding_window_size` (int) and `milestone_interval` (int) fields populated from the scenario's NegotiationParams
8. THE NegotiationStateModel Pydantic model SHALL include `milestone_summaries_enabled` (bool, default False), `milestone_summaries` (dict, default empty), `sliding_window_size` (int, default 3), and `milestone_interval` (int, default 4) fields for Firestore persistence
9. THE NegotiationStateModel SHALL accept documents from Firestore that do not contain the new fields without validation errors

### Requirement 3: Milestone Summary Generation

**User Story:** As a developer, I want a milestone summary generator that produces compressed strategic summaries at configurable turn intervals, so that agents retain strategic context from earlier turns without carrying full history.

#### Acceptance Criteria

1. WHEN `milestone_summaries_enabled` is True and the current `turn_count` is a non-zero multiple of `milestone_interval`, THE Milestone_Generator SHALL produce a Milestone_Summary for each agent in the scenario
2. THE Milestone_Generator SHALL make one lightweight LLM call per agent to generate each Milestone_Summary
3. THE Milestone_Generator LLM call SHALL use the same model configured for the agent being summarized (from `agent_config.model_id`)
4. THE Milestone_Generator prompt SHALL instruct the LLM to produce a summary capturing: key positions taken by all parties, major concessions made and received, unresolved disputes, regulatory concerns raised, and the overall trajectory of the negotiation
5. THE Milestone_Generator prompt SHALL include the agent's private reasoning context (inner thoughts, strategy) so the summary reflects the agent's own perspective
6. THE Milestone_Generator prompt SHALL include the full history up to the current turn and any existing milestone summaries for context
7. WHEN a Milestone_Summary is generated, THE Milestone_Generator SHALL store it in `milestone_summaries[agent_role]` as a dict with `turn_number` and `summary` fields
8. THE Milestone_Summary text output SHALL be constrained to a maximum of 300 tokens via the LLM `max_tokens` parameter
9. WHEN the LLM call for milestone generation fails, THE Milestone_Generator SHALL log the error and continue the negotiation without the summary (non-blocking failure)
10. THE Milestone_Generator SHALL track token usage from summary generation calls and add the count to `total_tokens_used` in the state

### Requirement 4: Full History Elimination from Prompts

**User Story:** As a developer, I want the prompt builder to stop sending full raw history to the LLM after the first milestone summary is generated, so that token cost per turn is bounded and predictable regardless of negotiation length.

#### Acceptance Criteria

1. WHEN `milestone_summaries_enabled` is True and at least one Milestone_Summary exists for the current agent, THE Agent_Node `_build_prompt` function SHALL exclude all raw history entries except the sliding window
2. WHEN `milestone_summaries_enabled` is True and at least one Milestone_Summary exists, THE Agent_Node `_build_prompt` function SHALL include all stored Milestone_Summaries for the current agent in chronological order in the user message
3. WHEN `milestone_summaries_enabled` is True but no Milestone_Summary exists yet for the current agent (turns before the first milestone), THE Agent_Node `_build_prompt` function SHALL include the full history (same as spec 100 structured memory behavior)
4. THE sliding window SHALL use the `sliding_window_size` value from the negotiation state instead of the hardcoded value of 3 from spec 100
5. THE milestone summaries section in the prompt SHALL be formatted with clear labels (e.g., "Strategic summary as of turn N:") and placed between the structured memory section and the sliding window section
6. THE full negotiation history SHALL remain stored in the LangGraph state `history` field for the SSE stream and UI display — only the LLM prompt construction is affected

### Requirement 5: Frontend Milestone Summaries Toggle

**User Story:** As a user, I want a "Milestone Summaries" toggle in the Advanced Options section that is separate from the structured memory toggle, so that I can opt into milestone-based memory compression independently.

#### Acceptance Criteria

1. WHEN the Advanced_Options_Section is expanded, THE Arena_Selector SHALL display a "Milestone Summaries" toggle below the "Structured Agent Memory" toggle
2. THE "Milestone Summaries" toggle SHALL include a brief description: "Generate periodic strategic summaries to compress negotiation history and cap token usage for long negotiations"
3. THE "Milestone Summaries" toggle SHALL default to off (disabled)
4. WHEN the user enables the "Milestone Summaries" toggle, THE Arena_Selector SHALL automatically enable the "Structured Agent Memory" toggle if it is not already enabled
5. WHEN the user disables the "Structured Agent Memory" toggle, THE Arena_Selector SHALL automatically disable the "Milestone Summaries" toggle
6. THE "Milestone Summaries" toggle SHALL be visually disabled (grayed out, non-interactive) when the "Structured Agent Memory" toggle is off, with a tooltip or helper text indicating the dependency
7. WHEN the user selects a different scenario, THE Arena_Selector SHALL reset the "Milestone Summaries" toggle to off

### Requirement 6: Pass Milestone Toggle to Backend

**User Story:** As a developer, I want the milestone summaries enablement flag transmitted from the frontend to the backend when a negotiation starts, so that the orchestrator knows whether to generate milestone summaries.

#### Acceptance Criteria

1. WHEN the user clicks "Initialize A2A Protocol", THE Arena_Selector SHALL include a `milestone_summaries_enabled` boolean field in the StartNegotiationRequest payload
2. THE StartNegotiationRequest Pydantic model SHALL accept an optional `milestone_summaries_enabled` field of type `bool` with a default of `False`
3. WHEN the backend receives a StartNegotiationRequest with `milestone_summaries_enabled` set to True, THE negotiation router SHALL pass the value through to `create_initial_state`
4. WHEN the backend receives `milestone_summaries_enabled` as True, THE negotiation router SHALL also set `structured_memory_enabled` to True regardless of the `structured_memory_enabled` field value in the request (enforcing the dependency server-side)
5. WHEN the backend receives a StartNegotiationRequest without the `milestone_summaries_enabled` field, THE negotiation router SHALL default to `False`

### Requirement 7: Milestone Generation Orchestration Integration

**User Story:** As a developer, I want milestone summary generation to be triggered at the correct turn intervals within the LangGraph orchestration flow, so that summaries are available before the next agent turn begins.

#### Acceptance Criteria

1. WHEN the dispatcher node detects that `turn_count` is a non-zero multiple of `milestone_interval` and `milestone_summaries_enabled` is True, THE dispatcher SHALL trigger milestone summary generation before routing to the next agent
2. THE milestone generation step SHALL execute after the turn count increments (on cycle wrap in `_advance_turn_order`) and before the next agent node runs
3. WHEN milestone generation completes, THE dispatcher SHALL merge the updated `milestone_summaries` into the state before proceeding
4. WHEN milestone generation fails for one agent, THE dispatcher SHALL continue generating summaries for remaining agents and proceed with the negotiation
5. THE milestone generation step SHALL be skipped entirely when `milestone_summaries_enabled` is False (zero overhead)

### Requirement 8: Token Budget Predictability

**User Story:** As a system operator, I want token cost per turn to be bounded and predictable after the first milestone summary, so that I can forecast LLM costs regardless of negotiation length.

#### Acceptance Criteria

1. WHEN `milestone_summaries_enabled` is True and at least one Milestone_Summary exists, THE prompt sent to the LLM for an agent turn SHALL contain at most: the system prompt, the agent's structured memory, all milestone summaries for that agent, the sliding window of raw messages, and current state instructions
2. THE combined token count of the prompt components listed in criterion 1 SHALL remain bounded as negotiation length increases — specifically, the only component that grows is the milestone summaries list, which grows at a rate of one summary (max 300 tokens) per `milestone_interval` turns
3. WHEN `milestone_summaries_enabled` is False, THE prompt builder SHALL behave identically to spec 100 behavior (structured memory + hardcoded 3-message sliding window when structured memory is enabled, or full history when structured memory is disabled)

### Requirement 9: Backward Compatibility

**User Story:** As a user running existing scenarios, I want the negotiation engine to behave identically when milestone summaries are disabled, so that no existing functionality is broken.

#### Acceptance Criteria

1. WHEN `milestone_summaries_enabled` is False, THE Agent_Node SHALL build prompts using the spec 100 behavior (structured memory + sliding window if structured memory is enabled, or full history if disabled)
2. WHEN `milestone_summaries_enabled` is False, THE dispatcher SHALL skip all milestone generation logic with zero performance overhead
3. WHEN the `milestone_summaries_enabled` field is absent from the NegotiationState (e.g., sessions created before this feature), THE system SHALL treat the value as False
4. WHEN the `sliding_window_size` or `milestone_interval` fields are absent from the NegotiationState, THE system SHALL use default values of 3 and 4 respectively
5. THE NegotiationStateModel SHALL accept documents from Firestore that do not contain `milestone_summaries_enabled`, `milestone_summaries`, `sliding_window_size`, or `milestone_interval` fields without validation errors
6. FOR ALL existing scenario JSON files, THE scenario loader SHALL parse them without errors when the new NegotiationParams fields are absent

### Requirement 10: Milestone Summary Serialization Round-Trip

**User Story:** As a developer, I want milestone summaries to serialize and deserialize cleanly through all state boundaries, so that summary data is never corrupted or lost in transit.

#### Acceptance Criteria

1. FOR ALL valid milestone summary dicts containing `turn_number` (int) and `summary` (str), storing in `milestone_summaries[role]` list, serializing to NegotiationStateModel, and reconstructing SHALL preserve all data without loss
2. THE milestone summary dicts SHALL contain only JSON-serializable types (no custom objects, datetimes, or non-primitive types)
3. FOR ALL valid NegotiationState dicts containing `milestone_summaries`, converting to NegotiationStateModel via `model_dump()` and back SHALL produce equivalent milestone summary data
